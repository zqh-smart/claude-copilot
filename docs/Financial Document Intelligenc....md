**Financial Document Intelligence System** (Living Design Document)

***

### 🧭 系统目标

构建一个企业级金融文档智能系统，实现：

* **多源文档解析**：高效处理年报、财报、公告、行业研究报告。
* **结构化金融知识构建**：将非结构化文档转化为可计算、可检索、可推理的知识单元。
* **深度投研分析**：支持公司基本面、行业对比、增长潜力等分析。
* **主动风险识别**：自动识别财务、经营、市场、政策等多维度风险。
* **自动报告生成**：一键生成投资研报、风险分析报告、公司分析报告。
* **多公司对比分析**：支持跨公司、跨行业、跨时间维度的对比推理。

***

### 🏗 系统总架构 (v0.1)

系统采用分层架构，自下而上依次为：

1. **Data Ingestion Layer**：负责文档的收集、分类与预处理。
2. **Document AI Layer**：文档理解引擎，负责金融知识的抽取与结构化。
3. **Knowledge Layer**：混合知识系统，整合向量、图谱、关系型数据库。
4. **Reasoning Layer**：多Agent推理系统，模拟金融分析决策。
5. **Workflow Layer**：基于Dify的工作流编排与调度引擎。
6. **Application Layer**：产品化与商业化接口，提供多种终端应用。

各层之间通过标准化的数据接口连接，确保模块化与可扩展性。

## 一、Data Ingestion Layer (数据摄取层)

### 📌 输入数据类型

* 年度报告 (PDF，核心数据源)
* 财务报表 (Excel/PDF)
* 公司公告 (PDF/HTML)
* 行业研究报告 (PDF/DOC)

### ⚙️ 文档解析策略 (企业级多解析器架构)

采用 **Parser Router（解析器路由）** 作为核心，根据文档类型自动选择最优解析器：

* **标准PDF** → Unstructured (通用工业方案)
* **扫描/图像PDF** → PaddleOCR / Qwen-VL (多模态OCR)
* **复杂表格PDF** → Camelot / MinerU (表格专用)
* **Office文档** → LlamaParse / Dify原生解析器
* **HTML网页** → DOM清理 + Readability提取

### 📦 解析工具栈 (2026最佳实践)

* **Docling**：结构化解析，语义切分精准。
* **MinerU**：处理复杂版式PDF，表格恢复能力强。
* **Unstructured**：工业级标准方案，稳定性高。
* **LlamaParse**：针对RAG场景优化，上下文保留完整。
* **Marker**：PDF转Markdown/JSON，轻量高效。
* **Apache Tika**：通用后备解析器，确保兼容性。

### 📤 输出标准结构

所有解析后的文档统一为以下JSON格式，为下游处理提供一致接口：

```json
{
  "doc_id": "唯一标识符",
  "company": "公司名称",
  "year": "报告年份",
  "sections": ["文本段落数组"],
  "tables": ["表格数据数组"],
  "metadata": {
    "type": "annual_report",
    "source": "文件来源",
    "parsed_at": "时间戳"
  }
}
```

### 📌 Dify在本层的角色定位

Dify在本层主要负责 **工作流编排与调度**，而非核心解析逻辑。它将多个解析节点串联，管理数据流转、异常处理与状态追踪。

## 二、Document AI Layer (文档理解引擎 v0.2)

> **📌 核心定位：金融知识结构化引擎 (Financial Knowledge Structuring Engine)**&#x672C;层的任务不是简单的PDF解析或文本切分，而是将非结构化金融文档转化为 **“可计算 + 可检索 + 可推理”的结构化金融知识单元**。

### 🧩 系统子模块拆解

#### **2.1 Document Understanding Pipeline (文档理解流水线)**

```text
输入: 原始PDF文档
↓
Layout Detection (版面检测)
↓
Semantic Segmentation (语义切分)
↓
Entity Recognition (实体抽取)
↓
Structure Reconstruction (结构重建)
↓
Financial Schema Mapping (金融模式映射)
↓
输出: 结构化知识对象
```

#### **2.2 Layout Understanding (版面理解层)**

* **目标**：识别文档的**视觉结构**，而非纯文本流。
* **识别对象**：
  * 标题层级 (H1, H2, H3...)
  * 表格区域与边界
  * 图表区域与说明
  * 风险提示段落 (如“风险因素”章节)
  * 财务摘要块 (如“合并利润表”)
* **技术方案**：

| **场景** | **推荐技术**         |
| ------ | ---------------- |
| 通用场景   | Unstructured     |
| 高精度要求  | Docling          |
| 多模态理解  | GPT-4o / Qwen-VL |
| 学术研究   | LayoutLMv3       |

#### **2.3 Semantic Segmentation (语义切分层) - 核心升级点**

替代传统的Token切分，基于金融文档语义进行智能切分：

* **切分逻辑**：
  * 经营情况讨论与分析块
  * 财务表现分析块
  * 风险因素披露块
  * 管理层讨论与分析 (MD\&A) 块
  * 行业与竞争环境分析块
* **输出结构**：

```json
{
  "segment_id": "seg_001",
  "segment_type": "risk_section",
  "content": "具体文本内容...",
  "confidence": 0.93,
  "page_range": [5, 7]
}
```

#### **2.4 Financial Entity Extraction (金融实体抽取 ⭐核心)**

* **抽取对象**：
  * **🏦 财务指标实体**：营收、净利润、ROE、毛利率、现金流等。
  * **🏢 企业实体**：公司名称、子公司、关联方、控股结构。
  * **⚠️ 风险实体**：市场风险、信用风险、流动性风险、政策风险。
  * **📅 事件实体**：并购、分红、高管变动、监管处罚。
* **技术方案**：
  * **LLM结构化抽取**：采用GPT-4o、DeepSeek-V3等模型，结合Schema约束解码。
  * **规则后处理**：基于金融词典与正则表达式进行验证与补全。
  * **多模型投票**：提升关键财务数字抽取的准确率。

#### **2.5 Financial Schema Mapping (结构化映射 ⭐关键)**

* **核心任务**：将抽取的实体与文本，映射到统一的、标准化的金融数据结构。
* **标准Schema示例**：

```json
{
  "company": "公司A",
  "year": 2023,
  "financial_metrics": {
    "revenue": 100000000,
    "net_profit": 15000000,
    "operating_cash_flow": 18000000,
    "growth_rate": 0.12
  },
  "business_segments": [
    {"name": "软件销售", "revenue": 60000000}
  ],
  "risk_factors": [
    {"type": "market_risk", "description": "..."}
  ],
  "events": ["完成B轮融资"],
  "subsidiaries": ["子公司X"]
}
```

* **关键价值**：统一的数据结构是后续进行GraphRAG、多公司对比分析的**基础前提**。

#### **2.6 Table Intelligence Engine (财务表格引擎)**

财报中超过60%的关键信息（尤其是数值）存在于表格中。

* **处理流程**：

```text
PDF中的表格
↓
表格检测 (Camelot / MinerU)
↓
结构恢复与单元格合并
↓
列对齐与表头识别
↓
语义映射 (判断是损益表、资产负债表等)
↓
数据归一化并与主Schema合并
```

* **输出结构**：

```json
{
  "table_id": "tbl_001",
  "table_type": "income_statement",
  "rows": [...],
  "normalized_metrics": {
    "revenue": 932420978,
    "cost_of_revenue": 111000000
  }
}
```

#### **2.7 Risk Intelligence Extractor (风控智能抽取模块)**

* **功能**：自动识别并分类文档中披露的各类风险。
* **输出结构**：

```json
{
  "risk_id": "risk_001",
  "risk_type": "market_volatility_risk",
  "description": "受宏观经济波动影响，公司主营业务收入存在不确定性...",
  "severity": "high", // low, medium, high
  "source_section": "risk_disclosure",
  "extracted_from": "page_12_paragraph_3"
}
```

#### **2.8 Document Fusion Engine (多源文档融合)**

* **问题**：同一公司的年报、临时公告、券商研报可能存在数据冲突或信息互补。
* **解决方案**：
  * **时间优先级**：以最新披露文件为准。
  * **权威性评分**：年报 > 官方公告 > 第三方研报。
  * **置信度融合**：对不同来源抽取的同一指标进行加权平均。
* **输出**：生成该公司的 **统一知识主档案**。

### 🧠 本层核心升级总结

1. **从 Chunking 到 Semantic Structuring**：抛弃无意义的token切分，转向基于金融语义的智能切分与理解。
2. **从 Text RAG 到 Financial Schema RAG**：检索的不再是文本片段，而是结构化的金融数据对象，极大提升推理精度。
3. **从 PDF 到 Knowledge Object**：输出不再是文本，而是携带类型、关系、置信度的**可计算金融知识对象**。
4. **从 Extraction 到 Financial Understanding**：流程集成了分类、映射、融合，为深度推理做好了前置结构化准备。

### 🔗 层间关系

* **输入**：来自 Data Ingestion Layer 清洗后的标准化文档。
* **输出**：结构化的金融知识单元，直接输入 Knowledge Layer 的三类数据库。

> **本质总结**：Document AI Layer 是系统的 **“金融认知建模层”**，它构建了机器理解公司业务、财务与风险的**数据基础**。

## 三、Knowledge Layer (混合知识层 v0.3)

> **📌 本层目标：构建统一的金融知识系统，支撑多文档检索、结构化计算、跨公司推理、风险传播分析与行业对比。**

### 🧩 整体架构：Hybrid Knowledge System

系统采用三层知识存储与一个智能调度中枢的架构：

```text
                    ┌──────────────────────┐
                    │   Query Interface    │
                    └─────────┬────────────┘
                              ↓
               ┌────────────────────────┐
               │  Retrieval Orchestrator │ (智能路由与调度)
               └──────┬───────┬─────────┘
                      ↓       ↓
        ┌────────────────┐ ┌────────────────┐
        │   Vector DB    │ │    Graph DB    │
        │ (语义检索层)   │ │ (关系推理层)   │
        └────────────────┘ └────────────────┘
                      ↓       ↓
               ┌────────────────────────┐
               │      SQL Layer         │
               │ (结构化数值层)         │
               └────────────────────────┘
                              ↓
               ┌────────────────────────┐
               │     Fusion Engine      │ (多源结果融合)
               └────────────────────────┘
                              ↓
               ┌────────────────────────┐
               │   LLM Reasoning Layer  │
               └────────────────────────┘
```

### 🧠 三层知识体系详解

#### **3.1 Vector Layer (语义检索层)**

* **作用**：处理基于文本语义的模糊查询，提供RAG基础能力。
  * 示例问题：“公司管理层对未来发展有何展望？”
* **技术实现**：
  * **嵌入模型**：text-embedding-3-large, BGE-M3, Qwen Embedding。
  * **向量数据库**：Qdrant (推荐)，Milvus (超大规模场景)。
* **数据结构**：

```json
{
  "chunk_id": "vec_001",
  "text": "原始文本",
  "embedding": [0.1, 0.2, ...],
  "metadata": {
    "company": "公司A",
    "year": 2023,
    "section": "management_discussion",
    "doc_id": "doc_xyz"
  }
}
```

* **查询方式**：语义搜索、混合搜索 (BM25 + 向量)。

#### **3.2 Graph Layer (知识图谱层 ⭐核心升级)**

* **为什么必须引入图谱**：金融分析的核心是**关系推理**，而非文本匹配。
  * 例如：分析“公司A的供应链风险如何传导至公司B？”
* **图谱结构设计**：
  * **节点类型 (Nodes)**：
    * `Company` (公司)
    * `BusinessSegment` (业务板块)
    * `FinancialMetric` (财务指标)
    * `Risk` (风险因子)
    * `Industry` (行业)
    * `Event` (事件)
  * **边类型 (Edges / Relationships)**：
    * `OWNS` (拥有) - 公司拥有子公司
    * `OPERATES_IN` (经营于) - 公司经营某项业务
    * `HAS_RISK` (存在风险) - 公司面临特定风险
    * `AFFECTED_BY` (受...影响) - 风险受政策影响
    * `COMPETES_WITH` (与...竞争) - 公司间的竞争关系
    * `SUPPLIES_TO` (供应给) - 供应链上下游关系
* **图谱示例**：

```
(Company:指南针)-[OPERATES_IN]->(Industry:金融信息服务)
(Company:指南针)-[HAS_RISK]->(Risk:市场波动风险)
(Company:指南针)-[OWNS]->(Company:子公司A)
(Company:指南针)-[COMPETES_WITH]->(Company:同花顺)
```

* **技术选型**：Neo4j (推荐，生态完善)、NebulaGraph (分布式超大规模)、NetworkX (轻量级原型)。
* **查询能力**：支持多跳查询、路径发现、风险传播链分析、社区发现。

#### **3.3 SQL Layer (结构化金融数据层)**

* **作用**：处理精确的数值计算、聚合与对比。
  * 示例问题：“公司A过去三年的营收复合增长率(CAGR)是多少？行业平均水平是多少？”
* **表结构设计核心表**：
  * **Company (公司主表)**：`company_id`, `name`, `industry`, `listed_exchange`。
  * **FinancialMetrics (财务指标表)**：`record_id`, `company_id`, `year`, `revenue`, `net_profit`, `roe`, `cash_flow`。
  * **Risks (风险表)**：`risk_id`, `company_id`, `risk_type`, `risk_level`, `description`, `year`。
* **SQL计算能力**：CAGR计算、同比增长率、环比增长率、行业分位数排名、财务比率分析 (如流动比率、负债率)。

### 🧠 Retrieval Orchestrator (检索调度器)

* **功能**：智能分析用户查询意图，动态路由至最合适的知识源，并融合结果。
* **查询流程**：

```text
用户查询
↓
Query Analyzer (意图分类)
↓
Routing Decision (路由决策):
  ├── 语义/描述性问题 → Vector DB
  ├── 关系/影响链问题 → Graph DB
  ├── 精确数值/计算问题 → SQL DB
  └── 复杂混合问题 → 并行查询三者
↓
Result Fusion (结果融合)
↓
送入LLM进行最终推理与回答
```

### 🧠 Fusion Engine (融合引擎)

* **输入**：来自向量、图谱、SQL三个独立检索通道的原始结果。
* **输出**：统一、消歧、互补的增强型上下文，提供给LLM。
* **输出结构**：

```json
{
  "query_intent": "risk_analysis",
  "retrieved_context": {
    "vector_snippets": ["相关文本段落1", "..."],
    "graph_paths": ["公司->风险->政策"],
    "sql_metrics": {"revenue_growth": 0.15, "industry_avg": 0.10}
  },
  "fusion_summary": "整合后的分析要点..."
}
```

### 🚀 GraphRAG升级 (核心趋势)

将传统RAG升级为GraphRAG，实现从“检索文本”到“检索与推理关系”的飞跃。

| **维度**   | **传统RAG** | **GraphRAG**    |
| -------- | --------- | --------------- |
| **信息单元** | 文本片段      | 结构化实体与关系        |
| **推理能力** | 单跳，基于文本   | 多跳，基于关系路径       |
| **典型应用** | 问答、总结     | 风险传播、竞争分析、影响链推演 |

### 🧠 本层核心创新与本质

1. **三层融合**：非简单并列，而是构建了 **“金融混合知识系统”**，语义、关系、数值能力互补。
2. **智能路由**：Retrieval Orchestrator 实现了查询意图到知识源的动态最优匹配。
3. **关系推理**：Graph Layer 的引入使系统具备了回答复杂关系问题的能力。
4. **本质**：Knowledge Layer 是系统的 **“金融认知推理核心大脑”**，它从“数据库”升级为“推理系统”。

***

## 四、Reasoning Layer (多Agent推理层 v0.4)

> **📌 本层目标：将“知识检索系统”升级为“金融决策支持系统”，通过多智能体协作模拟专业分析团队的推理过程。**

### 🧩 总体架构

采用 **Router -> Decomposer -> 并行Agent执行 -> 聚合与生成** 的协作流水线。

```text
          用户查询
             ↓
    ┌──────────────────┐
    │   Router Agent   │ (意图分类)
    └────────┬─────────┘
             ↓
 ┌─────────────────────────┐
 │ Task Decomposer Agent   │ (任务拆解)
 └──────────┬──────────────┘
            ↓
 ┌─────────────────────────────────────┐
 │     并行多Agent系统                 │
 │  ├── Research Agent    (投研分析)   │
 │  ├── Risk Agent        (风险识别)   │
 │  ├── Quant Agent       (财务计算)   │
 │  ├── Comparator Agent  (对比分析)   │
 │  └── Critic Agent      (自检纠错)   │
 └──────────┬───────────────────────────┘
            ↓
    ┌──────────────────┐
    │ Result Aggregator│ (结果融合)
    └────────┬─────────┘
             ↓
    ┌──────────────────┐
    │ Final Generator  │ (报告生成)
    └──────────────────┘
```

### 🤖 核心Agent设计

#### **4.1 Router Agent (路由智能体)**

* **输入**：原始用户查询。
* **功能**：识别查询意图（投研、风控、对比、计算等），并初步判断问题复杂度。
* **输出**：结构化任务描述，引导下游拆解与分发。

#### **4.2 Task Decomposer Agent (任务拆解智能体)**

* **输入**：Router Agent的输出。
* **功能**：将复杂问题（如“分析公司A的全面风险”）拆解为可并行执行的原子子任务。
* **示例**：问题“分析指南针2023年风险”可拆解为：`[财务风险提取, 经营风险提取, 市场风险分析, 合规风险检查]`。

#### **4.3 Research Agent (投研分析智能体) ⭐核心**

* **职能**：模拟卖方分析师，进行公司基本面与投资价值分析。
* **输入**：Knowledge Layer提供的融合上下文（财务数据、业务描述、行业信息）。
* **推理方法**：链式思维 (CoT)，结合金融分析框架（如SWOT、波特五力）。
* **输出**：结构化的投资分析观点，包括成长驱动因素、财务健康度、投资建议（正面/中性/负面）。

#### **4.4 Risk Agent (风险识别智能体) ⭐关键**

* **职能**：模拟风控专员，系统化识别、评估与分类风险。
* **输入**：风险相关的文本、图谱风险节点、历史风险数据。
* **策略**：规则引擎（识别特定关键词）与LLM判别相结合。
* **输出**：风险清单，包含类型、描述、严重等级（高/中/低）、可能影响。

#### **4.5 Quant Agent (财务计算智能体)**

* **职能**：专精于精确数值计算，**避免LLM在计算上的幻觉与误差**。
* **特点**：本质是“工具调用Agent”，接到计算任务后，自动编写并执行SQL查询或调用计算引擎。
* **输出**：精确的计算结果，如CAGR、ROE、估值比率（PE/PB）。

#### **4.6 Comparator Agent (对比分析智能体)**

* **职能**：执行跨公司、跨行业、跨时间维度的对比分析。
* **输入**：多个实体的知识上下文。
* **输出**：对比矩阵，突出差异点、相对优势与劣势。

#### **4.7 Critic Agent (自检与纠错智能体) ⭐企业级关键**

* **职能**：扮演“质量检查员”，对上游多个Agent的初步结果进行逻辑一致性、数据完整性与合理性审查。
* **输出**：问题清单与置信度评分，必要时触发特定Agent重新执行或补充分析。

### 🔄 多Agent协作机制

1. **并行执行**：Decomposer拆解出的子任务，由对应Agent并行执行，提升效率。
2. **Critic反馈回路**：初步结果必经Critic审查，形成自我改进闭环。
3. **聚合与升华**：Aggregator Agent将各专项结果融合，生成全面、平衡的综合分析。
4. **结构化生成**：Final Generator 依据强模板，将结构化分析转化为易读的报告或回答。

### 🧠 本层核心创新与本质

1. **从单一模型到分析团队**：用多Agent系统模拟了投研团队（分析师、风控员、计算员、复核员）的协作。
2. **任务拆解与并行**：破解了单一LLM处理复杂任务时逻辑易混乱的难题。
3. **工具增强**：通过Quant Agent将计算任务卸载给专业工具，保证数值精确性。
4. **自我批判与改进**：Critic机制显著提升了系统的可靠性与稳健性。
5. **本质**：Reasoning Layer 是系统的 **“金融决策模拟系统”**，实现了从检索到深度分析与决策支持的跨越。

## 五、Workflow Layer (Dify工作流编排层 v0.5)

> **📌 本层目标：将前四层的AI能力（RAG、Graph、Agent）转化为可工程化、可产品化、可稳定执行的标准化业务流程。**

### 🎯 Dify的核心定位

在本系统中，Dify **不是AI核心**，而是 **“AI流程操作系统”**，负责：

* **节点化编排**：将复杂流程拆解为可复用、可监控的节点。
* **状态管理**：维护工作流执行状态，处理异常与重试。
* **资源调度**：协调LLM调用、工具执行、数据查询等资源。
* **API封装**：将内部复杂流程暴露为简洁的API端点。

### 🧩 五大核心工作流设计

系统根据不同业务场景，预设多条标准化工作流。

#### **5.1 智能问答工作流 (Q\&A Workflow)**

* **目标**：处理事实型、描述型快速问答。
* **流程**：`用户查询 → 意图路由 → 混合检索(Vector/Graph/SQL) → 上下文融合 → LLM生成回答`。
* **特点**：低延迟，强依赖检索质量，不触发复杂Agent。

#### **5.2 投研分析工作流 (Investment Research Workflow) ⭐核心**

* **目标**：生成深度的公司或行业分析报告。
* **流程**：`用户查询 → 路由 → 并行知识检索 → Research Agent分析 → Critic审查 → 报告生成`。
* **输出**：结构化的投资分析摘要，或完整的研报草稿。

#### **5.3 风险分析工作流 (Risk Analysis Workflow) ⭐企业关键**

* **目标**：系统化识别与评估目标主体的风险。
* **流程**：`用户查询 → 路由 → 风险任务拆解 → Graph风险路径检索 → Risk Agent分析 → 风险报告生成`。
* **特点**：对图谱检索依赖度最高，输出要求高度结构化。

#### **5.4 对比分析工作流 (Comparison Workflow)**

* **目标**：执行两个或多个实体间的多维度对比。
* **流程**：`用户查询 → 实体提取 → 多实体并行检索 → SQL数值对比 → Graph关系对比 → Comparator Agent生成对比矩阵`。

#### **5.5 自动报告生成工作流 (Report Generation Workflow) ⭐商业核心**

* **目标**：一键生成可供下载或分发的正式报告（PDF/Markdown）。
* **触发**：可由用户手动触发，或按计划（如季度末）自动运行。
* **流程**：`选定公司/行业 → 批量文档检索 → 财务数据聚合 → 投研与风险Agent并行分析 → 结果融合 → 报告模板渲染 → 文件导出`。
* **输出**：符合行业规范格式的正式报告文档。

### ⚙️ 工作流设计原则

1. **状态机设计**：工作流非简单线性，支持条件分支、循环、回退、人工审核节点。
2. **Agent封装**：所有Agent均被包装为标准化工作流节点，接受统一输入/输出规范。
3. **分层检索**：在工作流中显式区分并调用Vector、Graph、SQL三种检索能力。
4. **结构化输出**：每个工作流的最终输出必须是结构化的数据或文档，确保下游系统可处理。

### 🧠 本层核心价值与本质

1. **产品化引擎**：将前沿AI技术封装成稳定、可靠、可配置的业务流程。
2. **可视化运维**：通过Dify界面，产品经理和运维人员可直观监控、调整流程。
3. **本质**：Workflow Layer 是系统的 **“金融AI产品操作系统”**，是实现工程化落地的关键桥梁。

## 六、Application Layer (应用产品层 v0.6)

> **📌 本层目标：将AI能力转化为最终用户可感知、可交互、可商业化的产品与服务。**

### 🧩 产品形态矩阵

系统通过多种前端形态服务不同用户角色。

#### **6.1 智能对话分析系统 (Chat Analysis System) - 核心入口**

* **形态**：类ChatGPT的交互界面。
* **功能**：用户通过自然语言提问，系统调用后端工作流，返回综合答案（含公司概览、财务快照、风险提示、来源引用）。
* **用户**：所有用户，特别是投资分析师。

#### **6.2 自动报告生成系统 (Report Generation System) - 核心商业产品**

* **形态**：报告生成中心。
* **功能**：用户选择公司、时间范围、报告类型（投研/风控），系统自动生成并交付格式规范的报告（PDF/HTML），支持下载。
* **价值**：直接替代或辅助人工撰写报告，是明确的付费点。

#### **6.3 对比分析平台 (Comparison System)**

* **形态**：专用对比工具界面。
* **功能**：用户拖选多家公司或行业，系统生成多维对比看板（财务对比图、风险雷达图、业务重叠度）。

#### **6.4 BI可视化仪表盘 (BI Dashboard)**

* **形态**：数据可视化Dashboard。
* **功能**：展示关键财务指标趋势图、风险热力图、行业分布图、公司排名榜。
* **用户**：管理层、投资决策者，用于宏观概览与趋势洞察。

### 🔌 API服务体系

为第三方集成与企业级调用提供标准化API。

* **Chat API** (`POST /api/v1/chat`): 提供智能问答服务。
* **Report API** (`POST /api/v1/report`): 触发报告生成，异步返回报告文件。
* **Data API** (`GET /api/v1/companies/{id}/metrics`): 提供结构化财务与风险数据。

### 👥 用户角色与用例

* **👤 投资分析师**：深度使用Chat与Report系统，进行快速调研与报告撰写。
* **👤 风控专员**：聚焦Risk Analysis Workflow与风险Dashboard，监控风险暴露。
* **👤 管理层/投资经理**：使用BI Dashboard把握整体portfolio状况与行业趋势。

### ☁️ 部署与商业化

* **部署架构**：
  * **MVP版**：单体Docker Compose部署，适合概念验证与小团队。
  * **企业版**：基于Kubernetes的微服务架构，数据库与AI服务独立扩展，支持高可用。
* **商业化模式**：
  * **SaaS订阅**：按分析的公司数量、文档处理量、API调用量分级收费。
  * **私有化部署**：为大型金融机构提供本地化部署、数据隔离的专属解决方案。
  * **API服务**：将核心能力（如财报分析、风险识别）打包为API，赋能其他金融科技应用。

### 🚀 最终系统总结

**Financial Document Intelligence System** 通过六层架构，实现了从原始金融文档到智能决策支持的完整闭环：

1. **Data Ingestion**：解决了“文档如何进来”的问题。
2. **Document AI**：解决了“文档如何被理解”的问题。
3. **Knowledge Layer**：解决了“知识如何存储与关联”的问题。
4. **Reasoning Layer**：解决了“如何像专家一样思考”的问题。
5. **Workflow Layer**：解决了“如何工程化执行”的问题。
6. **Application Layer**：解决了“如何交付价值”的问题。

**最终本质**：这是一个 **“金融文档认知与决策支持操作系统”** ，旨在将非结构化的海量金融文本，转化为可行动的结构化知识与智能，赋能投资研究、风险管理与企业决策。
